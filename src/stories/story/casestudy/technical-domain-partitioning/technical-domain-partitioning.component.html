How to organize code

How should I organize my source files? What partitioning strategy would be the correct one? Most architecturally conscious developers and teams ask these questions before starting to write code for a new project. I surely do, every time. The answer, as for many questions in software architecture, is that it depends.

The starting point is usually a layered architecture. It has grown to be such a significant pattern in modern software development that layered architecture can be considered a universal architectural origin at this point. In this pattern, logic is organized into responsibility groups or layers. Every type of software can define its standard layer stack, grouping similarly working mechanics. Once a layer stack is defined, there is only a single common rule to enforce: layers can only depend on each other in a single direction. This keeps complicated code robust, clean, easier to interpret, and quicker to navigate. Consequently, the project gains better developer experience and becomes more maintainable.

I believe the most popular standard layer stack is of the MVC (Model-View-Controller) web application type. Since MVC applications are monolithic software with frontend and backend components, the layer stack is infrastructure, models, services, factories, controllers, and views. On the other hand, a frontend application has fewer responsibilities, and therefore, its standard layer stack revolves around presentation. An Angular application is built from services, components, directives and pipes. Using custom layer stacks is also common, especially in Domain-Driven Design, where layers are rather domain-specific than application-specific.

Technical vs domain partitioning

In a layered architecture, there are two popular strategies for organizing code: technical partitioning and domain partitioning. While there are no formal definitions for them, by summarizing books and articles about software architecture, we can use the following approximations:

  • Technical partitioning: popularized by clean architecture, source files are located in folders named after layers in the application's standard layer stack.
  • Domain partitioning: popularized by Domain-Driven Design, source files are grouped by functionality. Folder names represent domain modules in the application.

Technical versus domain partitioning in architecture. Source: Neal Ford et. al. (2022): Software Architecture: The Hard Parts

It has been bothering me from the beginning that the two major partitioning strategies are always framed as oppositions. Choose one over the other. Not only in books and articles, but frameworks also tend to vote for one against the other. Most frameworks advocate technical partitioning, since they can provide CLI generators for adding new components to standard layers. Others, like Angular, are putting their feet down with domain partitioning. Such preferences tend to propagate quickly, as people follow official style guides. Consequently, open-source code bases acting as references for new projects reflect the same structure.

This is a major anti-pattern in my opinion. We should treat partitioning strategies like other software design choices: as patterns. Frameworks should stop making preferences and focus on showcasing best practices for different situations in their style guides. Experience tells me that there are simple cases where pure technical partitioning is the best choice, but in most cases, using a mix of the two is the way forward. I usually choose a strategy based on the following flowchart.

My usual workflow of choosing a partitioning strategy

The flowchart contains 4 combinations from the 2 strategies as possible choices:

  • Technical: a pure technical partitioning pattern. Only viable if the application is small enough to have a single responsibility. Single-purpose microservices or applications with the size of a company landing page come to mind.
  • Technical over domain: technical first, then domain-specific subfolders. If you don't know the final complexity or don't have the resources to make proper plans, this is a stable mix with good flexibility. Prepare for a large refactor in the next stage, though. It does not scale well with complexity.
  • Domain over technical: domain first, but technical subfolders can exist. Flexible for uncertainties in the business logic, but keeps the code base clean and scales well with complexity. You need to plan, as it takes considerable effort to change the base structure later.
  • Domain: a pure domain partitioning pattern. Only viable if you have mature ubiquitous language and a well-defined domain model of the real-life business logic the software needs to aid or automate. No technical folders required, as layers are domain-specific. Great for large enterprises with clear vision, mature workflows, vast know-how, and experienced and interconnected teams.

Subject at hand, a.k.a. the case in the study

When I first created the website you are currently browsing, I planned to add a few pages dedicated to showcasing skills offered by my company. To keep the architecture's complexity in check, I used a pure technical partitioning strategy. Then came the stories part as a lightweight blog system. Additionally, I added some tools related to spatial visualization techniques. All in all, I ended up with a steep increase in overall complexity. To prevent the application from spiraling out of control, I added some domain folders, transitioning to a "technical over domain" mixed strategy.

After a while, the application still ended up as architectural spaghetti. Two new domain concepts emerged: stories and web mapping. However, most components still lived under flat technical folders, like components, pages, directives, and layouts. To restore maintainability and extensibility, I decided to bite the bullet and rework the technical over domain strategy to a domain over technical one.

To illustrate how even a frontend application's architectural refactoring offers multiple viable decisions, there would have been an alternative. In such a case, you can decide to solve the issue by decomposition. Done by separating domains and creating micro-frontends from them, integrating all the pieces into a central layout. For me, that would have been implausible, as I deploy the website to GitHub pages as a single deployment unit.

First phase: defining principal domains

My goal in the initial phase was to define a rough domain structure in the codebase and to eliminate flat technical folders, such as components, directives, models, pages, pipes, services, and utils. The last commit in the old structure is 31c8980.

State

  • Software elements are mostly in flat, technical folders.
  • The two existing domains (stories, web-mapping) are just folders.
  • The dependency graph is chaotic. It's challenging to navigate the codebase due to its current complexity.

I applied an incremental, step-by-step restructuring technique, where I identified coarse-grained domains and moved together source files exclusively used by them. This step was a bunch of file-moving and namespace-renaming operations. Most of the restructuring affected the original company pages, since stories and spatial visualization had their own domain folders from the beginning.

In the end, every company page got its own domain and owned all exclusive components, models, utilities, and services. The repository state at that point can be browsed at 8b06eea. This step was fairly trivial, but the need for fine-tuning was obvious. The code base got littered with shared folders. Those annoying shared folders! I really cannot stand them, as they don't fit in my mental models. They are not technical or domain-specific; they are shared, a common functional attribute in clean object-oriented code. I often use the term for projects, libraries, and system components, but not for software elements.

State

  • Software elements are grouped in coarse-grained domains.
  • The dependency graph reflects ownership, it's easier to navigate.
  • There are a lot of shared folders, weakening the structure.
  • Edge cases (like array.ts) are just placed somewhere.

Second phase: refinement by planning

The resulting structure after the first phase was strong enough to call it a structure and not chaos. However, it was still not strong enough to hit the bar. Edge case utilities were living in random places. There was a "layouts" technical partition I had no idea what to do with. Domain boundaries were not clear enough to make the structure really logical and easily maintainable.

For the next step, moving files between folders was not enough. To go further, I needed to create a software plan. My favorite method for software planning is the C4 model. It is a simple, extendable model capable of mapping a software system on different levels, like an online global map. If you zoom out, you see continents. As you zoom in, details are continuously unfolding before you until you see your house.

The first level represents the context of the whole software system. For a simple frontend application without many coupling points, it's not too interesting. The second level unwraps the software system and displays coupling between its components. That is also very straightforward, although there are some interesting details about the build pipeline.

Level 1 (left) and level 2 (right) C4 plans of this website

In this case, the real magic happens on the third level. It contains the internal workings of software in a software system. That is the level to reason about structure and to move every element to its final place. Another iterative task. I picked a coarse-grained domain, identified every source file associated with it, and made one of the following decisions:

  • Keep the source file in its current partition.
  • Move the source file to another partition where it's a better fit.
  • Create a new partition for the source file and move it there.

Using this method, domains were clarified one by one. New subdomains emerged, as well as technical partitions inside domains where a technical grouping could be justified. Edge cases also found their places in the refined structure. It turned out that layout is a perfectly fine domain term in a frontend application. Components defining the common layout of pages are layout components. Once I finished with every major domain, I ended up with a large but really informative C4 level 3 diagram.

Level 3 C4 plan of this website

Purple rectangles are routed components, meaning they are used by the router to compose pages. Blue rectangles are elements used in routed components. Grey rectangles are libraries or other external dependencies. Blue dashed contexts are domain partitions. Green dashed contexts are technical partitions. Not every file is displayed; there is no need for that in such a plan, only the important ones affecting the application's structure.

While the plan seems large and convoluted, once you get the grasp of it, you can progress really fast. I managed to complete the 3 levels in less than 8 hours. The diagram reflects commit 7137b38, although the same structure is preserved in newer versions.

State

  • Software elements are grouped in fine-grained domains and technical partitions.
  • Dependent components are close and easy to navigate.
  • No shared folders, no edge cases, everything is in its place.

As a bonus, the C4 plan shed some light on architectural flaws. Fixing them along with code smells and other issues will be the topic of a future case study, though.

  • The sandbox page is a hedgehog with high efferent coupling and therefore high instability.
  • The feature support page and the map page depend on the same set of components, showing a lack of abstraction. That part of the plan is a maze.
  • The sandbox page's navigation bar depends on a faraway layout component to provide its own customization. Long arrows moving across multiple partitions are hard to track.
